# 手动拓展

本文说明如何在 OpenCode Pocket Kit 中手动添加工具密钥、环境变量、Skills、项目级配置和 npm 插件。

## 基本原则

OpenCode Pocket Kit 的便携环境遵循以下原则：

- 密钥通过 `编辑密钥.cmd` 加密保存，不明文写入配置文件。
- OpenCode 主配置位于 `config\opencode\opencode.json`。
- OMO 路由配置由 Oh-My-OpenAgent 官方安装器生成，通常位于 `config\opencode\oh-my-openagent.json`。
- 全局 Skills 放在 `config\opencode\skills\`。
- 项目级 Skills 放在 `项目目录\.opencode\skills\`。
- npm 插件优先安装到 `config\opencode\`，避免写入系统全局环境。
- 手动改配置前，建议先备份对应文件。
- 修改配置后，建议运行 `诊断.cmd` 检查。

## 添加 Tavily API Key

Tavily 不是模型供应商，不需要写入 `opencode.json` 的 `provider`。

正确做法是运行：

```bat
编辑密钥.cmd
```

然后加入：

```env
TAVILY_API_KEY=tvly-xxxx
```

保存后通过以下方式启动：

```bat
启动.cmd
```

或者在项目目录中运行：

```bat
opkcode
```

启动脚本会解密 `vault\secrets.env.enc`，并把 `TAVILY_API_KEY` 注入当前 OpenCode 进程。Tavily Skills 或相关工具只要读取环境变量，就可以使用这个密钥。

## 添加其他环境变量

`编辑密钥.cmd` 支持任意 `KEY=VALUE` 形式的环境变量。

例如：

```env
DEEPSEEK_API_KEY=sk-xxxx
OPENROUTER_API_KEY=sk-or-v1-xxxx
TAVILY_API_KEY=tvly-xxxx
GITHUB_TOKEN=ghp_xxxx
CUSTOM_TOOL_TOKEN=xxxx
```

注意：

- 每行一个变量。
- 不要在变量名前后加多余空格。
- 不要把 API Key 写入 `opencode.json`。
- 不要把明文 `.env` 长期放在 U 盘中。
- 如果需要修改密钥库，重新运行 `编辑密钥.cmd` 并输入完整变量列表。

## 模型供应商和工具密钥的区别

模型供应商通常需要 OpenCode provider 配置，例如：

- DeepSeek
- OpenRouter
- OpenAI
- SiliconFlow
- Moonshot
- DashScope
- Zhipu

工具密钥通常只需要环境变量，例如：

- Tavily：`TAVILY_API_KEY`
- GitHub：`GITHUB_TOKEN`
- 其他搜索、浏览器、自动化工具的 token

OpenCode Pocket Kit 默认使用 OMO 路由能力，不强制在 `opencode.json` 中写入模型供应商。需要精细控制 provider 时，可以手动编辑 `opencode.json`。

## 手动添加 OpenCode provider

如果你要手动添加 provider，可以编辑：

```text
config\opencode\opencode.json
```

示例结构：

```json
{
  "$schema": "https://opencode.ai/config.json",
  "plugin": [
    "oh-my-openagent"
  ],
  "provider": {
    "example": {
      "npm": "@ai-sdk/openai-compatible",
      "name": "Example Provider",
      "options": {
        "baseURL": "https://example.com/v1",
        "apiKey": "{env:EXAMPLE_API_KEY}"
      },
      "models": {
        "example-model": {
          "name": "Example Model"
        }
      }
    }
  },
  "model": "example/example-model"
}
```

注意：

- 这里的 `EXAMPLE_API_KEY` 应通过 `编辑密钥.cmd` 写入密钥库。
- `baseURL` 和模型名必须以供应商官方文档为准。
- 不要删除 `"plugin": ["oh-my-openagent"]`，否则 OMO 不会加载。
- 修改后运行 `诊断.cmd`。

## 手动安装全局 Skills

全局 Skills 目录是：

```text
config\opencode\skills\
```

一个 Skill 通常是一个文件夹，里面至少包含：

```text
SKILL.md
```

示例结构：

```text
config\opencode\skills\my-skill\
└─ SKILL.md
```

安装步骤：

1. 创建目录：

   ```text
   config\opencode\skills\my-skill\
   ```

2. 放入 Skill 文件：

   ```text
   config\opencode\skills\my-skill\SKILL.md
   ```

3. 重启 OpenCode：

   ```bat
   启动.cmd
   ```

或者在项目目录中运行：

```bat
opkcode
```

## 手动安装项目级 Skills

项目级 Skills 只对当前项目生效。

示例：

```text
D:\Code\my-project\.opencode\skills\my-skill\
└─ SKILL.md
```

适合这些情况：

- 只想让某个项目使用特定 Skill。
- 不希望全局 Skills 目录变得复杂。
- 需要在项目仓库中一起管理项目专用规则。

使用命令行入口时，可以这样操作：

```bat
cd D:\Code\my-project
opkcode
```

OpenCode 会以当前目录作为工作目录，并读取项目内 `.opencode` 配置。

## 从 GitHub 手动复制 Skill

有些仓库会把多个 Skills 放在一个 `skills` 目录下，例如：

```text
skills\frontend-design\SKILL.md
skills\pdf\SKILL.md
```

你只需要复制目标 Skill 文件夹，而不是整个仓库。

例如把：

```text
skills\frontend-design\
```

复制到：

```text
config\opencode\skills\frontend-design\
```

最终应形成：

```text
config\opencode\skills\frontend-design\SKILL.md
```

如果多套了一层目录，例如：

```text
config\opencode\skills\frontend-design\frontend-design\SKILL.md
```

OpenCode 可能无法按预期发现。应把内层目录内容上移。

## 手动安装 npm 插件

OpenCode 插件通常是 npm 包。

建议安装到便携配置目录：

```bat
npm install --prefix config\opencode 插件包名
```

例如：

```bat
npm install --prefix config\opencode some-opencode-plugin
```

安装后，编辑：

```text
config\opencode\opencode.json
```

在 `plugin` 数组中添加插件名：

```json
{
  "$schema": "https://opencode.ai/config.json",
  "plugin": [
    "oh-my-openagent",
    "some-opencode-plugin"
  ]
}
```

注意：

- 不建议把插件安装到系统全局 npm。
- 不要删除已有的 `oh-my-openagent`，否则 OMO 不会加载。
- 插件名必须与插件包要求的加载名一致。
- 修改后运行 `诊断.cmd` 检查。

## 手动卸载 npm 插件

如果插件安装在便携配置目录，可以运行：

```bat
npm uninstall --prefix config\opencode 插件包名
```

然后从：

```text
config\opencode\opencode.json
```

中删除对应插件项。

不要删除：

```json
"oh-my-openagent"
```

除非你明确不再使用 OMO。

## 手动修改 OpenCode 主配置

主配置文件是：

```text
config\opencode\opencode.json
```

最小推荐结构：

```json
{
  "$schema": "https://opencode.ai/config.json",
  "plugin": [
    "oh-my-openagent"
  ]
}
```

`plugin` 数组一般应包含字符串插件名。不要写成：

```json
{
  "plugin": [
    {
      "Length": 15
    }
  ]
}
```

如果出现类似对象，说明配置被错误脚本或编辑过程污染，应恢复为字符串数组。

## 手动修改 TUI 配置

TUI 配置文件是：

```text
config\opencode\tui.json
```

推荐至少包含：

```json
{
  "plugin": [
    "oh-my-openagent/tui"
  ]
}
```

如果 OMO Doctor 提示 TUI plugin missing，可以检查这个文件。

## OMO 路由配置

OMO 路由配置通常由官方安装器生成，常见位置是：

```text
config\opencode\oh-my-openagent.json
```

也可能使用以下文件名：

```text
config\opencode\oh-my-openagent.jsonc
config\opencode\oh-my-opencode.json
config\opencode\oh-my-opencode.jsonc
```

建议优先保留官方安装器生成的配置，不要手动生成空文件覆盖它。

如果你想手动调整 OMO 的模型路由，应先备份：

```text
config\opencode\oh-my-openagent.json
```

然后再编辑对应字段。

## 项目级 OMO 配置

某个项目可以有自己的 OMO 配置：

```text
项目目录\.opencode\oh-my-openagent.json
```

或：

```text
项目目录\.opencode\oh-my-openagent.jsonc
```

项目级配置适合：

- 某个项目使用不同的 agent 路由。
- 某个项目使用特定模型。
- 某个项目需要独立工具规则。

## 手动安装后如何检查

运行：

```bat
诊断.cmd
```

重点检查：

- `opencode.json` 是否存在。
- `tui.json` 是否存在。
- OMO 插件是否已启用。
- OMO Doctor 是否还有问题。
- Comment Checker 是否可用。
- 相关工具链是否能被找到。

如果只是添加了新的环境变量，可以重新运行：

```bat
编辑密钥.cmd
启动.cmd
```

或：

```bat
opkcode
```

## 常见问题

### Tavily 配了但不能用

检查密钥库中是否包含：

```env
TAVILY_API_KEY=tvly-xxxx
```

然后确认是通过以下入口启动：

```bat
启动.cmd
```

或：

```bat
opkcode
```

不要直接双击 `bin\opencode.exe`，否则密钥库不会被解密，环境变量也不会被注入。

### 手动添加的 Skill 不显示

检查目录结构是否正确：

```text
config\opencode\skills\skill-name\SKILL.md
```

常见错误是多套了一层目录：

```text
config\opencode\skills\skill-name\skill-name\SKILL.md
```

这种情况下需要把内层 `skill-name` 提到外层。

### 插件安装了但不生效

检查三点：

1. 插件是否安装在：

   ```text
   config\opencode\node_modules\
   ```

2. `opencode.json` 是否包含插件名：

   ```json
   "plugin": [
     "oh-my-openagent",
     "your-plugin-name"
   ]
   ```

3. 是否通过 `启动.cmd` 或 `opkcode` 启动，而不是直接运行 `bin\opencode.exe`。

### 修改配置后启动异常

可以先备份当前配置，然后恢复最小配置：

```json
{
  "$schema": "https://opencode.ai/config.json",
  "plugin": [
    "oh-my-openagent"
  ]
}
```

再运行：

```bat
诊断.cmd
```

## 建议的拓展顺序

建议按这个顺序拓展：

1. 先运行 `安装.cmd`。
2. 运行 `诊断.cmd`，确保基础环境正常。
3. 运行 `编辑密钥.cmd`，加入模型供应商和工具密钥。
4. 运行 `启动.cmd` 或 `opkcode`，确认基本功能正常。
5. 再手动添加 Skills。
6. 最后再安装额外 npm 插件。

这样可以避免把基础环境问题和手动拓展问题混在一起。
